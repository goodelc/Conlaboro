## 1. 背景与决策依据

### 1.1 为什么需要管理端

| 驱动因素 | 说明 |
|----------|------|
| 内容治理 | 用户可能发布违规项目/评论，需要快速下架能力 |
| 用户管理 | 封禁恶意用户、调整角色、重置密码 |
| 运营统计 | 了解平台增长趋势、用户活跃度、项目完成率 |
| 系统健康 | 监控异常数据、处理反馈 |

### 1.2 为什么不在 MVP 阶段做

- 当前种子用户仅 16 人，出现问题直接操作数据库更高效
- Phase 2 尚有更高优先级的核心功能（CreatePage 对接、JoinModal 对接、OAuth 等）
- MVP 目标是验证「核心协作流程跑通」，不是运营效率

### 1.3 何时启动

**触发条件**（满足任一即可）：
- 注册用户数 ≥ 50
- 出现首例需要人工干预的违规内容
- 运营团队正式介入

## 2. 现状分析：权限体系完全缺失

### 2.1 当前权限模型

```
数据库 users.role  →  "admin" / "lead" / "member"  (VARCHAR(30))
         ↓
    但整个系统完全不读取这个值
         ↓
    JWT Token: 只携带 { username, userId, color }
    JwtAuthFilter: 硬编码 ROLE_USER
    SecurityConfig: 只有 permitAll / authenticated 两级
    前端 RequireAuth: 只检查 isLoggedIn
    前端 AppContext: 虽存储 role 但从未使用
```

### 2.2 需要改造的文件清单

| 层级 | 文件 | 当前问题 | 改造内容 |
|------|------|----------|----------|
| **JWT** | `JwtTokenProvider.java` | 不含 role | `generateToken` 加入 role claim |
| **JWT** | `JwtAuthFilter.java` | 硬编码 `ROLE_USER` | 从 token 读取 role，动态构建 authorities |
| **Security** | `UserPrincipal.java` | 无 role 字段 | 增加 `role` 属性 |
| **Security** | `SecurityConfig.java` | 无角色鉴权 | 添加 `/api/admin/**` → `hasRole('ADMIN')` |
| **DB** | `V1__Create_tables.sql` | 无 CHECK 约束 | 新增迁移脚本添加 role CHECK |
| **前端** | `RequireAuth.jsx` | 只检查登录 | 扩展为支持角色检查 |

```sql
UPDATE users SET role = 'member' WHERE role NOT IN ('admin', 'lead', 'member');
```

##### (3) JWT Token 注入 role

**`JwtTokenProvider.java`** 改造：

```java
// 修改前
public String generateToken(String username, Long userId, String color) { ... }

// 修改后
public String generateToken(String username, Long userId, String color, String role) {
    return Jwts.builder()
            .subject(username)
            .claim("userId", userId)
            .claim("color", color)
            .claim("role", role)          // 新增
            .issuedAt(now)
            .expiration(expiry)
            .signWith(getSigningKey())
            .compact();
}

// 新增
public String getRole(String token) {
    return parseToken(token).get("role", String.class);
}
```

##### (4) JwtAuthFilter 动态权限

**`JwtAuthFilter.java`** 改造：

```java
// 修改前
List<SimpleGrantedAuthority> authorities = Collections.singletonList(
    new SimpleGrantedAuthority("ROLE_USER"));

// 修改后
String role = jwtTokenProvider.getRole(token);
String authority = "ROLE_" + role.toUpperCase();
List<SimpleGrantedAuthority> authorities = Collections.singletonList(
    new SimpleGrantedAuthority(authority));
```

**`UserPrincipal.java`** 改造：

```java
    private final String role;                    // 新增
    private final Collection<? extends GrantedAuthority> authorities;

    public UserPrincipal(Long id, String username, String color, String role,
                         Collection<? extends GrantedAuthority> authorities) {
        // ...

    public String getRole() { return role; }
}
```

##### (6) SecurityConfig 添加 admin 路由保护

**`SecurityConfig.java`** 在 `anyRequest().authenticated()` 前添加：


##### (7) AuthController 登录接口返回 role

确保 `POST /api/auth/login` 的响应中包含 `role` 字段（当前可能已包含，需验证）。

#### 3A.2 前端改造

##### (1) RequireAuth 扩展

**`components/RequireAuth.jsx`** 改造：

```jsx
// 修改后：支持可选角色检查
export default function RequireAuth({ children, role }) {
  const { isLoggedIn, currentUser } = useApp()
  if (!isLoggedIn) return <Navigate to="/login" replace />
  if (role && currentUser.role !== role) return <Navigate to="/" replace />
  return children
}

// 使用方式

```

## 4. 关键设计决策

### 4.1 前端路由策略：嵌套 vs 独立

**选择：嵌套路由 `/admin/*`**

理由：
- 复用现有布局（Navbar、Toast、认证体系）
- admin 用户同时也是普通用户，需要自由切换
- 避免维护两套独立的布局系统

### 4.2 封禁机制：硬删除 vs 状态标记

**选择：状态标记（`status` 字段）**

理由：
- 封禁是可逆操作，用户可能被误封
- 保留数据完整性，不破坏外键关联
- 审计需要历史数据

### 4.3 审计日志：独立表 vs 嵌入现有表

**选择：独立审计表 `admin_audit_logs`**

理由：
- 审计日志是不可变记录，不应被业务操作修改
- 独立表便于查询和归档
- 不影响现有业务表结构

### 4.4 Admin 入口可见性

**选择：仅 admin 角色可见 Navbar 中的入口**

```jsx
// Navbar.jsx 中
{currentUser?.role === 'admin' && (
  <NavLink to="/admin">管理后台</NavLink>
)}
```

非 admin 用户无法看到入口，直接访问 URL 会被 `RequireAdmin` 拦截跳转。

## 5. 安全注意事项

### 5.1 后端防线

- **所有 `/api/admin/**` 接口必须经过 `hasRole('ADMIN')` 校验**（SecurityConfig 层面）
- **Controller 层二次校验**：通过 `@RequestAttribute("userId")` 获取当前用户，查询数据库确认其 role 为 admin
- **不可信任前端传参**：修改角色等操作，后端必须验证操作者自身也是 admin
- **敏感操作需要确认**：删除用户、删除项目等操作需要审计记录

### 5.2 前端防线

- `RequireAdmin` 组件同时检查 `isLoggedIn` 和 `role === 'admin'`
- API 层面返回 403 时，前端自动跳转首页并提示无权限
- 敏感操作（删除、封禁）使用确认弹窗，防止误触

### 5.3 JWT 刷新问题

**注意**：修改用户角色后，该用户的现有 JWT 仍包含旧 role。处理策略：

1. **即时生效**：修改角色后要求用户重新登录（logout + 清除 token）
2. **渐进生效**：后端每次请求都从数据库查 role 而非仅依赖 JWT（性能换安全）

**推荐方案**：JWT 中存 role 用于路由级权限判断 + 后端关键操作时二次查库验证。

## 附录 B：最低成本应急方案

如果 MVP 期间确实遇到紧急管理需求，可直接用以下方式处理：

```sql
-- 封禁用户（修改密码使其无法登录）
UPDATE users SET password_hash = '$2a$10$LOCKED_ACCOUNT' WHERE email = 'bad-user@example.com';

-- 修改角色
UPDATE users SET role = 'admin' WHERE email = 'promote@example.com';

-- 删除违规评论
DELETE FROM comments WHERE id = 123;

-- 删除违规项目
UPDATE projects SET deleted = 1 WHERE id = 456;
```

> 此方案仅适用于用户量极小的早期阶段。
