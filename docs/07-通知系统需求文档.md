# 07 — 通知系统需求文档

> **模块**: 通知推送 / 未读计数 / 已读管理  
> **关联页面**: 全局 (Navbar 铃铃 + NotificationPanel)  
> **关联组件**: `NotificationPanel`, `Navbar`  
> **关联 API**: `GET /api/notifications`, `GET /api/notifications/unread-count`, `PUT /api/notifications/{id}/read`, `PUT /api/notifications/read-all`  
> **数据表**: `notifications`  

---

## 1. 功能概述

通知系统是 Conlaboro 中连接用户与项目事件的桥梁，确保用户不会错过任何重要动态：

| 能力 | 说明 |
|------|------|
| **实时提醒** | 项目申请、评论、动态变更等事件触发通知 |
| **未读计数** | Navbar 显示未读数 badge，吸引用户点击 |
| **通知面板** | 右侧抽屉式面板，展示完整通知列表 |
| **已读管理** | 单条标已读 / 一键全部已读 |
| **分类展示** | 不同类型使用不同 icon，便于快速识别 |

## 3. 数据模型

### 3.1 notifications 表

```sql
CREATE TABLE notifications (
    id          SERIAL PRIMARY KEY,
    user_id     INT NOT NULL REFERENCES users(id),   -- 接收者
    type        VARCHAR(30) NOT NULL,                 -- 通知类型分类
    type_icon    VARCHAR(10),                        -- 类型图标 emoji
    title       VARCHAR(200) NOT NULL,                -- 通知标题
    content     TEXT,                                 -- 通知正文内容
    link        VARCHAR(500),                         -- 跳转链接 (可选)
    is_read      BOOLEAN DEFAULT FALSE,                  -- 是否已读
    created_at   TIMESTAMPTZ DEFAULT NOW()             -- 创建时间
);
```

### 4.3 交互逻辑

#### 打开面板
```js
function toggleNotif() {
  dispatch({ type: 'TOGGLE_NOTIF' })
  // reducer:
  // TOGGLE_NOTIF → { ...state, notifOpen: !state.notifOpen, notifCount: state.notifOpen ? 0 : state.notifCount }
  // 打开时清零计数（视觉上）
}
```

#### 关闭方式
1. 点击 X 关闭按钮
2. 点击遮罩层 overlay
3. 再次点击 🔔 按钮（toggle）

#### 标记已读
两种模式:

**单条已读**: 点击通知项 → 调用 `markAsRead(id)`
```js
async function handleItemClick(notif) {
  await markAsRead(notif.id)
  // 后续: 刷新通知列表或本地移除该项
}
```

**全部已读**: 面板底部或 header 区域提供「全部标已读」按钮
```js
await markAllAsRead()
```

#### 自动拉取未读数
```js
// 登录时 + 定期轮询 (Phase 3 WebSocket 替代)
useEffect(() => {
  if (isLoggedIn) fetchUnreadCount()
}, [isLoggedIn])

async function fetchUnreadCount() {
  try {
    const count = await getUnreadCount()
    dispatch({ type: 'SET_NOTIF_COUNT', count })
  } catch { /* 静默失败 */ }
}
```

## 5. API 接口

### 5.1 获取通知列表

```
GET /api/notifications
Authorization: Bearer {token}

Response (200):
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "type": "apply",
      "typeIcon": "📋",
      "title": "新的项目申请",
      "content": "Alex 申请加入「RecipeHub」的产品经理角色",
      "link": "/detail/0",
      "isRead": false,
      "createdAt": "2026-04-24T00:45:00Z"
    },
    ...
  ]
}
```

### 5.2 获取未读数量

```
GET /api/notifications/unread-count
Authorization: Bearer {token}

Response (200):
{
  "code": 200,
  "data": 3  // number
}
```

### 5.3 标记单条已读

```
PUT /api/notifications/{notificationId}/read
Authorization: Bearer {token}

Response (200):
{
  "code": 200,
  "message": "已标记为已读",
  "data": null
}
```

### 5.4 全部标为已读

```
PUT //api/notifications/read-all
Authorization: Bearer {token}

Response (200):
{
  "code": 200,
  "message": "全部已标记为已读",
  "data": null
}
```

## 6. 前端 API 封装 (`notification.js`)

```javascript
import request from './request'

export function getNotifications() {
  return request.get('/notifications')
}

export function getUnreadCount() {
  return request.get('/notifications/unread-count')
}

export function markAsRead(id) {
  return request.put(`/notifications/${id}/read`)
}

export function markAllAsRead() {
  return request.put('/notifications/read-all')
}
```

所有调用都通过 Axios 实例自动附加 Bearer token。

## 7. Toast 通知 vs Notification Panel 的分工

| 维度 | ToastContainer | NotificationPanel |
|------|---------------|-------------------|
| **用途** | 操作结果即时反馈 | 信息堆积查看 |
| **位置** | 右下角固定 | 右侧抽屉 |
| **生命周期** | 3 秒自动消失 | 持久到用户关闭 |
| **类型** | success / error / info | 按时间排序的列表 |
| **数量** | 同一时间可堆叠多条 | 全局唯一 |
| **示例** | "发布成功" / "密码错误" | "有人申请加入" / "评论了你的设计" |

两者互补：Toast 用于「操作确认」，Notification Panel 用于「信息回顾」。
