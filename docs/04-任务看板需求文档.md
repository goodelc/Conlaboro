# 04 — 任务看板需求文档

> **模块**: 任务管理 / 认领 / 状态流转 / XP 奖励  
> **关联页面**: `DetailPage` (TaskBoard 子组件)  
> **关联 API**: (Phase 2) `PUT /api/tasks/{id}/claim`, `PUT /api/tasks/{id}/status`  
> **数据表**: `tasks`, `milestones`  

---

## 1. 功能概述

任务看板是项目执行阶段的核心协作工具，采用经典的 **Kanban 三列看板** 设计：

| 列 | 状态 | 说明 |
|----|------|------|
| **待认领** | `open` | 无人负责的任务，任何团队成员可认领 |
| **进行中** | `progress` | 已有人认领，正在执行 |
| **已完成** | `done` | 任务完成，XP 已结算 |

**层级结构**: 项目 → 里程碑(Milestone) → 任务(Task)

## 2. 用户故事

| # | 角色 | 故事 | 验收标准 |
|---|------|------|----------|
| US-01 | 团队成员 | 我想看看有哪些任务可以做 | DetailPage 任务看板看到三列任务分布 |
| US-02 | 团队成员 | 我想认领一个任务 | 点击「认领此任务」→ 按钮变「✓已认领」→ +XP 提示 |
| US-03 | 团队成员 | 我想标记任务完成 | (Phase 2) 拖拽或按钮变更状态 |
| US-04 | 团队成员 | 我想知道完成任务的奖励 | 每个 TaskCard 显示 +XP 数值 |
| US-05 | 发起人 | 我想按里程碑规划任务 | CreatePage 步骤③ 定义 3 个里程碑及其任务 |

## 3. 数据模型

### 3.1 milestones 表

```sql
CREATE TABLE milestones (
    id          SERIAL PRIMARY KEY,
    project_id  INT NOT NULL REFERENCES projects(id),
    title       VARCHAR(200) NOT NULL,       -- 里程碑名称
    status      VARCHAR(20) DEFAULT 'pending', -- done / current / pending
    date        VARCHAR(50),                 -- 时间区间描述 "4月10日 - 4月20日"
    sort_order  INT DEFAULT 0                 -- 排序权重
);
```

### 3.2 tasks 表

```sql
CREATE TABLE tasks (
    id          SERIAL PRIMARY KEY,
    milestone_id INT NOT NULL REFERENCES milestones(id),
    name        VARCHAR(200) NOT NULL,        -- 任务名称
    status      VARCHAR(20) DEFAULT 'open',    -- open / progress / done
    assignee    VARCHAR(50),                   -- 负责人用户名 (NULL=待认领)
    xp          INT DEFAULT 10                 -- 完成奖励贡献值
);
```

**结构**:
```
┌─────────────────────────────────┐
│  {task.name}                     │  ← .tc-name (fw 700, fs 0.8rem)
│                                 │
│  [ avatar ] {assignee}  +{XP}  │  ← .tc-meta
```

- `.tc-name`: font-size 0.8rem, font-weight 700, margin-bottom 0.4rem
- `.tc-meta`: flex space-between, font-size 0.65rem, color warm-gray
- `.tc-assignee`: flex items center, gap 0.3rem
  - 无 assignee 时 color = red (表示「待认领」)
- `.ta-avatar`: 18px circle, bg = user color, font-size 0.4rem, fw 700
- `.tc-xp`: fw 700, color red, bg red-8%, padding 0.15rem 0.5rem, border-radius 8px
- `.tc-claim-btn`: width 100%, margin-top 0.5rem, padding 0.35rem, bg red, white color, border none, fw 700, fs 0.7rem, cursor pointer
  - hover: bg red-dark
- `.tc-claim-btn.claimed`: bg success, pointer-events-none (不可二次认领)

**交互逻辑** (当前前端实现):
```js
function handleClaim(e) {
  e.stopPropagation()
  setClaimed(true)                                    // 本地状态变更
  appToast(`已认领「${t.name}」· +${t.xp} XP`, 'success')  // Toast 提示
}
```

**显示条件**:
```jsx
{!t.assignee && !isDone && !claimed && (
  <button onClick={handleClaim}>认领此任务 · +{t.xp} XP</button>
)}
{!t.assignee && !isDone && claimed && (
  <button className="claimed">✓ 已认领</button>
)}
// 已完成 or 已有 assignee → 不显示按钮
```

## 5. 业务规则

### 5.1 XP 奖励体系

| 任务类型 | XP 范围 | 示例 |
|----------|---------|------|
| 调研/分析类 | 10~20 | 用户调研报告(+15), 竞品分析(+15) |
| 设计类 | 20~30 | 低保真原型(+20), 高保真设计稿(+25) |
| 开发类 | 15~30 | 首页开发(+20), API开发(+25), 联调(+15) |
| 测试/运维 | 10~20 | 功能测试(+15), Bug修复(+15) |
| 里程碑完成 | 30 | 正式上线(+30) |

### 5.2 认领规则

| 规则 | 说明 |
|------|------|
| 一人一任 | 同一时间一个任务只有一个负责人 |
| 任意认领 | 任何项目成员都可认领 open 任务（不限于对应角色） |
| 不可重复 | 已认领的任务不能再次认领 |
| 完成锁定 | done 状态的任务无法变更 |

### 5.3 状态流转

```
open ──(认领)──→ progress ──(完成)──→ done
                         │
                         └─(释放?)──→ open  (Phase 3 TODO)
```

## 6. API 接口 (Phase 2 - TODO)

### 6.1 认领任务

```
PUT /api/tasks/{taskId}/claim
Authorization: Bearer {token}

Request Body: (可为空，userId 从 token 获取)

Response (200):
{
  "code": 200,
  "message": "任务认领成功",
  "data": {
    "taskId": 1,
    "assignee": "currentUser",
    "status": "progress",
    "xpEarned": 20
  }
}
```

### 6.2 更新任务状态

```
PUT /api/tasks/{taskId}/status
Authorization: Bearer {token}

Request:
{
  "status": "done"  // | "progress" | "open"
}

Response (200):
{
  "code": 200,
  "message": "任务状态更新成功",
  "data": null
}
```

### 6.3 获取任务列表 (已在 /api/projects/{id} 中返回)

任务数据作为 `getProjectDetail` 响应的一部分返回，无需独立接口。

## 7. 前端数据转换

**TaskBoard 接收到的 milestones 数据可能是两种格式**:

**格式 A (结构化对象)** — 来自 API:
```json
{ "tasks": [{ "name": "...", "status": "open", "assignee": "Lina", "xp": 15 }] }
```

**格式 B (字符串数组)** — 来自种子数据:
```json
{ "tasks": ["用户调研报告", "竞品分析", "PRD文档"] }
```

**兼容处理** (DetailPage.jsx TaskBoard 内):
```js
if (!ms[0].tasks || typeof ms[0].tasks[0] === 'string') {
  return <p>该项目尚未创建详细任务。</p>
}
```

**扁平化处理**:
```js
const allTasks = ms.flatMap(m => m.tasks.map(t => ({ ...t, msStatus: m.status })))
const openTasks = allTasks.filter(t => t.status === 'open')
const progressTasks = allTasks.filter(t => t.status === 'progress')
const doneTasks = allTasks.filter(t => t.status === 'done')
```
